# Mechanics taxonomy

> **The conceptual model for every mechanic the d20-folio engine models.** Audience: any agent
> (human or AI) extending the schema or the SRD data. **The per-kind detail is co-located TSDoc in
> `src/lib/grants.ts`** (each arm of the `Grant` union documents its payload + evaluator merge rule) —
> that is the single source of truth for the kind list, its count, and its semantics; it can't drift
> from the code. This doc is the conceptual overview + the design rules + the "how to add a kind"
> recipe + the domain index that points INTO `grants.ts`. See `docs/ARCHITECTURE.md` for the layered
> architecture and the presenter seam.

---

## How the system works

A character's effective stats are **not** computed from prose. The SRD database stores every
mechanic-bearing fact as a typed `Grant` on the data file (race trait, feat, class feature, magic
item, subclass feature). At render time the sheet aggregates every grant the character is currently
receiving through `evaluateGrants(sources)` (`src/lib/grants.ts`) into `AggregatedGrants`, and reads
the aggregated view.

This is the **declarative-grant pipeline** — the single seam between data and rendering:

```
SRD data file              evaluateGrants            UI / sheet
─────────────────          ──────────────            ────────────
SrdRaceTrait        ─┐
SrdFeatData         ─┤    ┌──────────────┐         ┌─────────────┐
SrdClassFeatureData ─┼──▶ │ AggregatedGr │ ──────▶ │  Sheet      │
SrdMagicItemData    ─┤    │ ants         │         │  Header,    │
SrdSubclassInfo     ─┘    └──────────────┘         │  Abilities, │
                                                   │  Combat     │
                                                   └─────────────┘
```

**No module reads English prose with a regex** — that is the architecture's core law. Adding a
mechanic = adding a `Grant` discriminated-union variant + an evaluator branch + (where applicable) a
consumer; never a regex over text.

**Grant SOURCES** are assembled by `src/lib/resolve-grant-sources.ts → resolveAllGrantSources`:
features (class / feat / race / subclass), equipped + attuned magic items, chosen Eldritch
Invocations, learned Battle-Master maneuvers, chosen Metamagic, prepared spells that carry standing
effects, and the character's **background**. A background emits only its **idempotent, set-union-safe**
benefits as grants (`skill-proficiency` + `tool-proficiency`); its ASI + origin feat stay
creation-owned (non-idempotent — re-applying would double-count), and 2024 backgrounds grant no
languages.

**Provenance** — every grant tracks its source feature id, so the UI renders "Darkvision 60 ft (from
Elf)" rather than a bare value.

**Spell-data structured facts** (`SrdSpellData`, not Grants) carry the cast-time numerics both the
Spells card and the combat card read from ONE source (S12/S12b/G24): `damageDice` / `healDice` (the base
NdM), `healAddsCastMod`, `damageType(s)` / `damageChoice`. Two S12b shapes model spells whose effect is
NOT one roll: **multi-instance** (`instances` + `instancesPerUpcast`) — a spell that creates N SEPARATE
damage instances, each rolling `damageDice` on its own (Magic Missile's 3 darts at 1d4+1 each, +1 dart
per slot above 1st; Scorching Ray's 3 rays at 2d6 each, +1 ray per slot above 2nd). The shared pure
`spellInstanceCount(spell, castLevel)` resolves the count at the cast level (base at the spell's own
level); both surfaces render `N × {dice}` via `spells.multiInstance`, with the per-instance `damageDice`
kept intact so a flat damage rider folds per instance before the UI multiplies. **Upcast dice scaling**
(`damageDicePerUpcast`, S12c) — a leveled DAMAGE spell's per-slot-level dice increment (Fireball `"1d6"`
above 3rd, Inflict Wounds `"1d10"` above 1st, Vitriolic Sphere `"2d4"` above 4th). The shared pure
`scaleUpcastDice(spell, castLevel)` resolves the slot total (base count + increment × steps above base,
same die face; any flat tail like `"10d6+40"`'s `+40` preserved) — the SAME "scale the count" rule as
`scaleCantripDice` / `spellInstanceCount`. The cast modal's slot rows preview the scaled dice (the L5
Fireball row reads `10d6`); the spell card shows the bare base. A ray-count spell uses `instancesPerUpcast`
instead (its dice stay constant, its instance COUNT scales). **Upcast HEAL scaling** (`healDicePerUpcast`,
RA-07) — the heal-side twin: a leveled HEAL spell's per-slot-level heal-dice increment (Cure Wounds /
Prayer of Healing `"2d8"` above their own level, Healing Word `"2d4"`, Mass Healing Word / Mass Cure Wounds
`"1d8"`/`"1d4"`), resolved by the SAME `scaleUpcastDice` (called with `{ damageDice: healDice,
damageDicePerUpcast: healDicePerUpcast }`). The cast modal's slot rows preview the scaled heal beside the
scaled damage (Cure Wounds L3 row reads `6d8`); the spell card shows the bare base + mod. Fixed-amount heals
(Heal `70`, Mass Heal, Goodberry) and non-scaling leveled heals (Aura of Vitality, Regenerate) carry no
increment. **Dual damage instance**
(`secondaryDamage: { dice, damageType, dicePerUpcast? }`) — a SECOND simultaneous instance with its OWN
dice + type the single `damageType`/`damageDice` pair can't hold (Ice Storm 2d10 Bludgeoning + 4d6 Cold,
Ice Knife 1d10 Piercing + 2d6 Cold, Meteor Swarm 20d6 Fire + 20d6 Bludgeoning); both apply at once, the
surfaces append `+ {dice} {type}` after the primary (budget-gated), and it scales independently through
the SAME `scaleUpcastDice`. **Spell-area recurrence**
(`recurrence: SpellRecurrence`) — a stable token for a spell whose damage RE-APPLIES on a self-side
cadence (the spell analogue of the S3 feature duration cadence): `"on-enter-or-end-turn"` (a per-turn
area save — 2024 RAW "enters the area or ends its turn there"; Moonbeam, Spirit Guardians),
`"bonus-action-move"` (a hazard re-aimed as a Bonus Action —
Flaming Sphere), `"action-retrigger"` (a bolt re-fired as a Magic action — Call Lightning). The token
(ids only — golden rule 7) localizes to a cadence note on the spell card (a detail tag) and the
combat card (a gloss line) via `spells.recurrence_<token>`; informational only (no geometry, no dice —
golden rule 21). **Instantaneous duration** (`instantaneous: true`, B3) — the structured fact replacing
the old EN-prose `srdEn("spell", id, "duration") === "Instantaneous"` branch (golden rule 7: never
branch on a localized/prose string). Both consumers that drop the duration row for an instant spell (the
combat action summary in `smart-tracker`, the spell card in `spells-view`) read this boolean; the
localized duration DISPLAY string still lives in `src/i18n/<locale>/srd/spells.json` and resolves only at
the render boundary. Omitted (falsy) for any non-instantaneous spell — derived once from the canonical EN
duration at edit time, never re-parsed in code. **Costly material component**
(`components.costGp?` / `components.consumed?`, RA-23) — the STRUCTURED cost facts for a priced Material
(M) component: `costGp` is the FIRST/primary gp figure named in the SRD Components line (the compendium
prose keeps the full breakdown), `consumed` marks a spell that consumes it. Set together, only on
`m: true` spells, filled for the 53 gp-priced SRD spells and derived once from the canonical EN
Components prose at edit time (never re-parsed in code); omitted when the material has no gp cost, and
`consumed` is written `true` or omitted, never `false` (lean data, like `instantaneous`). The
`buildMaterialCostTag` edge helper formats a compact "M: 300 gp, consumed" chip
(`spells.materialCost`/`materialCostConsumed`, EN gp / IT mo) that leads the Spells-tab card's tag foot
— the surface you cast from; custom spells never match. The S12b **`diceByLevel`** extension reaches the level-scaled `aura` `ranged-attack` /
`heal` effect (Circle-of-Stars Archer/Chalice 1d8 → 2d8 at Druid 10) and the `form-attack` `damageDie`
(the Starry-Form Archer attack row) through the SAME shared "highest threshold ≤ level" helper
`pickDiceByLevel` the `ActionAttack.diceByLevel` resolver uses — one rule, three consumers (aura presenter
`auraVMs`, `resolveFormAttacks`, save-attack resolver). A **`form-attack`** (a transformation's
natural/built-in weapon — Wild Shape bite, Starry Archer, Armorer model weapon) is declared inside a
`while-active` block so it reaches `agg.formAttacks` ONLY while the form toggle is lit; `resolveFormAttacks`
computes its to-hit/damage from the character's scores (the mandated `attackAbility` reads EFFECTIVE
scores — B7 — so a Headband lifts the Armorer's INT). It can be DOUBLY gated by nesting a
`choice-grant-bundle` inside the `while-active` (the bundle inherits the wrapping `activeKey`): the
**Armorer's Arcane Armor** surfaces only the CHOSEN Armor Model's weapon (Dreadnaught Force Demolisher /
Guardian Thunder Pulse / Infiltrator Lightning Launcher) — donned-armor toggle AND model choice, switching
the model swaps the row. The **Circle-of-the-Stars Starry Form** uses the SAME doubly-gated shape: the
constellation chooser (`choice-grant-bundle` Archer/Chalice/Dragon) sits inside the form's `while-active`,
and the Archer ray's `form-attack` is nested in the `archer` OPTION (alongside its rail `aura`) — so the
WIS-keyed ranged Radiant attack row surfaces only while the form is lit AND Archer is chosen, and retracts
when you switch to the passive Chalice (heal aura) or Dragon (`roll-floor` aura) constellation. Two optional self-side riders ride the row: **`oncePerTurnExtra`** (`{ dice,
damageType }` → folds into the existing `summary.extraDamage` chip — the Infiltrator's once/turn +1d6
Lightning) and a catalogue-keyed on-hit **`note`** (presence-driven via `hasGrantField`, routed to
`summary.effect` — the Guardian's Disadvantage-on-others / Dreadnaught's push/pull reminder). The
reminder reaches the rendered weapon card through the ONE `buildWeaponFacts` seam as
`WeaponFactsVM.onHitNote` (the FULL localized sentence, not the 60-char collapsed-subtitle budget),
shown as an **"On a hit"** note in the expanded weapon-facts panel beside the rider strip — the same
channel the unarmed-strike "d8 if not holding a weapon" gloss now uses. All display-only (golden
rule 21), override-first via the shared weapon-swap store.

**Polymorph / True Polymorph SELF-swap** (S7) is a DISTINCT primitive from the `form-attack`
grant: a Beast form REPLACES the caster's whole stat block, so it is NOT a grant on the SRD data but a
**`BeastStatBlock`** (`src/data/types.ts`) — a self-contained CR-indexed record (id, `cr` fraction-capable,
`size`, `ac`, `hp`, `speeds`, all-six `abilityScores`, and FIXED `attacks` whose `toHit`+`damageDice` are
the Beast's PRINTED values, never owner-scaled — a `damageDice` may be a bare flat integer, e.g. `"1"`, for
the handful of CR-0 beasts that print no die). The forms live in the `src/data/beasts/` catalogue (ids +
numbers; names in the new **`beasts` `SrdKind`** catalogue, resolved via `localizeSrd`) — the FULL CR 0-8
Beast catalogue (91 forms): Phase 1's curated starter set of 18 iconic combat forms (CR 1/4→8) plus Phase
2's fill of 73 more forms spanning CR 0-6 (2026-07-07).
The pure `lib/polymorph.ts` exposes `resolvePolymorphForms(character)` (CR gate — form CR ≤ the caster's
level, the `BeastFormPicker`'s source) and `polymorphBuildPatch`/`polymorphPriorSnapshot`/`revertBuildFromPrior`
(the override-first applicator core). The store action `assumePolymorphForm(beastId, spellId)` STAMPS the
Beast's AC → `character.acOverride`, walk speed → `speedOverride`, other modes → `speedOverrides`, and ALL
six scores → `character.abilityScores` (RAW: Polymorph replaces every statistic, unlike Wild Shape which
keeps the mental scores), applies **Temp HP = the Beast's HP** (the `gainTempHp` max-wins seam), engages
Concentration by spell id, and records `session.polymorphForm = { beastId, spellId, prior }` (the `prior`
snapshot of the caster's own fields). Because every stamped value lands in a hand-editable override, a manual
edit still wins (override-first). The Beast's own attack rows render on the Play board via
`resolveBeastFormAttacks` — resolved DIRECTLY from the catalogue at the render edge (keeping `form-attack`
UNCHANGED), self-contained to-hit + dice. **The form ends on any of FOUR triggers**, each restoring the
body from `prior` + retracting the Temp HP + dropping the form (undoable): (1) a manual **Revert**
(`dropPolymorphForm()`); (2) **Temp-HP depletion** — 2024 RAW's PRIMARY end-trigger ("the spell ends early
on the target if it has no Temporary Hit Points left"): `applyDamage` ends the form the instant its Temp HP
reaches 0 (`newTemp === 0`), OUTRIGHT — no Concentration maintenance save is offered (the beat is routed
through the `concentration-dropped` toast, NOT the CON-save toast); (3) the caster dropping to **0 HP**
(subsumed by (2) — Temp HP absorbs first, so 0 real HP implies 0 Temp HP); and (4) a **Concentration drop or
swap** in `setConcentration` (casting another Concentration spell, or clearing concentration after a failed
real CON save) — the form's spell IS its sustaining Concentration, so ending that ends the form (the manual-
clear undo restores the whole pre-clear doc). Until the Temp HP is fully depleted a partial hit still prompts
the normal Concentration CON-save, and the stat-swap reaches that save **by construction** — the override-
carried Beast CON drives `applyDamage`'s effective-CON save. Polymorphing ANOTHER creature is a **read-only
reference card** in the picker (the app models ONE character — no second modeled entity, no override writes);
it renders the Beast's AC/HP/speeds/scores/attacks/senses/traits. Two accepted Phase-2 read-only limitations:
the reference CR gate uses the **caster's** level (the target's own level/size is not modeled), and — parity
with the Wild-Shape override-first speed limitation — a form's NON-walking speeds (fly/swim) are stamped into
`speedOverrides` but do not surface as NEW LeftHud speed rows on the SELF sheet (they DO show on the reference
card). `session.polymorphForm` is additive-only (absent on every non-polymorph doc → serialization unchanged;
the codec + sanitizer round-trip it).

## The activation + use-applies seams

A few cross-cutting behaviours ride the composite kinds; the per-kind TSDoc has the detail.

- **Activatable grants (`while-active`).** The one _wrapper_ kind: it nests any other grants behind a
  session toggle (`activeKey`). The evaluator merges the inner grants into the aggregate fields _only_
  while the key is active (`session.activeFeatures` ⇄ `toggleActiveFeature`), default off → never
  over-reports. Nesting is one level deep.
- **Activation seam.** A feature carrying BOTH `mechanics.actions` AND a `while-active` grant (Rage,
  Bladesong, Innate Sorcery, Starry Form) is INFERRED to be an activation feature: the smart-tracker
  stamps the resolved action with `activatesKey`, and the combat commit loop flips that key on use (the
  rail chip lights, the while-active grants flow into every derived figure, undo clears a commit-lit
  key). No feature is special-cased. A `while-active` BUFF SPELL is stamped the same way on its cast
  action — UNLESS the grant sets **`autoActivateOnCast: false`** (Warding Bond, a TARGET-ONLY buff:
  "you touch ANOTHER creature… it gains +1 AC and saves" — the CASTER never benefits): then no
  `activatesKey` is derived, casting never self-buffs, and the toggle stays MANUALLY light-able from
  the rail (override-first) — how a WARDED creature's own sheet turns the buff on. Omitted = auto-light
  (the self-buff default: Shield of Faith, Blur, Mage Armor).
- **USE-APPLIES seam.** Using an action AUTO-APPLIES its deterministic (dice-free) effects with the
  immediate-commit-with-undo model: a `temp-hp` grant carrying a `slot` (Orc Adrenaline Rush, Shifter
  Shifting, Chef) emits a resolved number applied on use; a `while-active.duration` lets the End-Turn
  loop surface a keep/end prompt for a maintained state (Rage), never silently killing it. Dice-bearing
  quantities (Second Wind's `1d10`) are NEVER auto-applied (golden rule 21).
- **Single-select chooser (`choice-grant-bundle`).** Pick exactly one of N named options; that
  option's grants apply (`session.grantBundleChoices` ⇄ `setGrantBundleChoice`). Option spells carry
  `minLevel` so the always-prepared injection level-gates them (Circle of the Land terrain spells).
- **Cadence engine (FRONTIER-S3 — per-turn recovery · round timers · round-1 clauses).** ONE generic
  turn/round engine in `smart-tracker.ts`, wired at the `TurnEconomyProvider.handleEndTurn` seam (run
  BEFORE the maintenance keep/end prompt — a hard expiry supersedes the soft prompt):
  - **Per-turn recovery** (`recovery: "per-turn"`). A tracker on this cadence auto-resets to full at the
    owner's turn start (Sneak Attack's once-per-turn use — never un-ticked by hand).
    `resolvePerTurnRecoveryTrackerIds` finds the spent ones; `characterStore.recoverPerTurnTrackers`
    resets them (undoable).
  - **Round timers** (`while-active.duration.maxRounds`). A `maintained` OR `timed` state declaring a
    `maxRounds` cap (Rage = 10) arms an `effectTimers[activeKey]` countdown when it lights
    (`armEffectTimers`), which `advanceEffectTimers` decrements each End Turn and AUTO-DROPS at 0 (clears
    the toggle, logs an `effect-expired` event, fires a toast). The chip shows "N rounds left". Persisted
    in the OPTIONAL `session.effectTimers` (absent = no timers; back-compat). All undoable via the single
    End-Turn undo. **Consumed buff potions** (S9) reuse the SAME map without a `while-active` source: a
    potion carrying `durationRounds` arms a self-sustaining `potion:<itemId>` timer on drink
    (`characterStore.consumePotionBuff`, undoable) that `advanceEffectTimers` carries forward + expires
    exactly like a state timer; the rail shows it as an "Active Potions" banner. Override-first — the
    buff's STATS are never auto-applied.
  - **Round-1 clauses** (`advantage-on { round1: true }`). A clause that applies only in combat round 1
    (Assassinate's first-round attack advantage); the consumer gates it on `combatStore.round === 1`,
    then it auto-clears.
- **The damage-and-dying flow (RA-03/05/10/11, 2026-07-12).** The player enters what their physical
  dice said; the engine computes the consequences (golden rule 21 — no RNG, ever):
  - **Damage intake** (`lib/damage-intake.ts`, pure). An entered hit is one or more
    `DamageInstance`s (`amount` + optional `type`/`source`); `resolveDamageIntake` applies the
    character's OWN effective defenses per SRD order — flat reduction → resistance halve (round
    down, ONCE — a type resistance and a source resistance never stack) → vulnerability double;
    immunity zeroes. `deriveDamageDefenses` (`lib/views/sheet-view.ts`) assembles the effective sets
    from the SAME seams the defenses rail renders (grants + the #68 build override maps + the
    PLAY-NO-EDIT session overlay + the gated flat reductions), so math and chips can never disagree.
    The HP popover offers type chips ONLY for defended types (minimum interaction), previews the
    math live with the same functions the commit applies, and an untyped amount passes verbatim
    (override-first). The store receives the NET total.
  - **0-HP rules** (`characterStore.applyDamage(amount, { crit })`). Crossing to 0 = a fresh dying
    state (track reset 0/0) + the Unconscious condition (SRD "Falling Unconscious"; shed by the
    heal-from-0 seam in `setHP` and by the at-zero "drop to 1 instead" interrupt); remainder past
    temp + current ≥ effective max = massive-damage instant death (3 failures — the ONE derived
    death predicate, `character-status.ts`). Damage while AT 0 never lowers HP: it marks one
    death-save failure (crit = two), ends a Stable state (successes clear), and damage ≥ max kills
    outright. Dead (3 failures) = inert.
  - **Death-save roll entry** (`useHpControls.applyDeathSave` consuming `compute.deathSaveOutcome`,
    incl. the Champion-Survivor `deathSaveCritThreshold`). The DyingBanner takes the entered d20:
    nat 1 = two failures, 2–9 = failure, 10+ = success (three = Stable), threshold+ = regain 1 HP
    and wake through the ONE heal-off-0 seam. The banner label is the verdict register
    (Dying → Stable → Dead); the pips remain directly tappable (override-first).
  - **Reversal.** Every applied consequence rides ONE undo entry; the reverse seam is
    `characterStore.restoreHpSnapshot` (HP + temp + dying track + conditions in one log-free,
    persisting write).
- **The Hide-check outcome seam (RA-12, 2026-07-21).** The 2024 Hide action rides a structured
  `summary.skillCheck` (`{ dc: 15, skill: "stealth" }`, ids only) on the `base-hide` card; the
  card's roll-entry takes the player's REAL d20 face (golden rule 21), folds the live Stealth
  bonus from the one shared skills derivation (`deriveSavesAndChecks`), and on a success
  `characterStore.applyHiddenState(total)` gains the Invisible condition + records
  `session.hiddenDc` — the SRD "your check total is the DC to find you" — in one undoable unit.
  The find-DC renders as a suffix on the rail's Invisible chip; `removeCondition("invisible")`
  clears it (undo restores both). A failed check mutates nothing (a plain notice). `hiddenDc`
  rides the parent doc while `invisible` lives in the `combat/state` subdoc (D9), so the
  trio-hydration seam `combat-state.applyCombatToSession` normalizes the pair: when the hydrated
  conditions no longer include `invisible` (e.g. a DM cleared it via a subdoc-only path) the
  orphaned find-DC is dropped — never at parse/sanitize time, where the trio is stripped from
  the parent doc.

## Cross-cutting design rules

1. **Grants always carry provenance.** `evaluateGrants` returns the merged effect AND the source id,
   so the UI can render "Darkvision 60 ft (from Elf)".
2. **Conflict resolution.** Same-kind grants on one character resolve by the kind's documented merge
   rule (highest-numeric for numerical kinds; union for set-kinds). The player can always override.
3. **Override-first, automation-by-default.** Every derived value (AC, PB, spell DC, HP max, tracker
   totals, attack count, fly speed, …) computes from grants by default; every value is manually
   overridable (`overrides` on the character / `trackerOverrides` on the feature ref). Surface the
   computed value; show "override" as a small affordance under it.
4. **Show formulas, never roll.** `Math.random()` is banned. The UI surfaces deterministic formulas
   ("1d8 + CHA", "DC = max(10, ⌊dmg/2⌋)"); the player rolls externally.
5. **One source of truth per fact.** A feature's scaling die lives on its `Grant` (or `levels[]`),
   never on a side map duplicated elsewhere.
6. **Italian never empty.** The i18n source cascade (CLAUDE.md i18n rule) applies to every
   user-visible string. SRD display strings live ONLY in the id-keyed catalogues
   (`src/i18n/{en,it}/srd/`); `src/data/**` is ids + mechanics only.
7. **Per-class level resolution.** A CLASS feature's level-scaling effect (Extra Attack count, Martial
   Arts die, a `damage-rider`'s `diceByLevel`) resolves at the character's level IN that owning class —
   RAW for a multiclass build, identical to the total for a single-class one. The forbidden
   anti-pattern is looking up `getClassTable(primaryClassId(c))` at `totalLevel(c)` for a class-specific
   value. **Spell save DC / spell attack bonus are PER SPELL** (2024 SRD 5.2.1 multiclassing): the
   owning class is derived from the spell's `classes[]` membership (`lib/spell-owning-class.ts`), so a
   Cleric/Wizard's Guiding Bolt uses WIS and their Fireball INT — one character, two DCs; a per-spell
   `spellAbilityOverride` (Magic Initiate) or a global DC/attack override wins (override-first).
8. **Multiclassing facts live on the class tables (#36).** The 13+ prerequisite reads `primaryAbility`
   - `primaryAbilityMode`; the partial "As a Multiclass Character" proficiency set is the table's
     `multiclass` field. The gate is `src/lib/multiclass.ts` (prerequisite both ways; illegal classes
     FILTERED, never greyed); armor/weapon training stays DERIVED (`featGateCtx`). Verified against
     dnd2024.wikidot.com. Full detail: `docs/ARCHITECTURE.md` (2024 multiclassing).

## How to add a Grant kind

1. **Add the variant** to the `Grant` discriminated union in `src/lib/grants.ts`, with co-located
   TSDoc stating its payload + evaluator merge rule (this doc's domain index needs no edit unless a new
   _domain_ appears).
2. **Add the evaluator branch** — one case in the `switch` inside `evaluateGrants`.
3. **Add (or extend) an `AggregatedGrants` field** the branch writes into.
4. **Add the consumer + a test.** Wire a consumer (sheet header, abilities page, combat panel, …) that
   reads from `AggregatedGrants` — never from prose — and pin the branch with a unit test (table-driven
   where a per-entity family exists). Then emit the grant from every SRD data row that should carry it
   (this step may be its own per-category release).

Steps 1-4 are usually one commit. `docs/AUTOMATION_COVERAGE.md` tracks per-entity coverage; the
override always remains (golden rule 8).

## Non-automatable residuals (the deliberate "did NOT automate, and why")

> A mechanic here is **not a gap** — it is a triaged decision. Re-auditing should treat each as
> already-settled. (The live-campaign frontier view is `docs/AUTOMATION_BACKLOG.md`; this is the
> standing taxonomy record.) Most former entries have since SHIPPED as PRIM grant kinds — those are
> noted in git / `CHANGELOG.md` and are NOT residuals.

The structural reasons the declarative-grant model is intentionally not the right shape:

- **Per-cast / per-attack choices.** Effects chosen at the moment of a single cast or attack (Paladin
  Sacred Weapon's 1-minute weapon enchant via Channel Divinity; Diviner Third Eye / "choose one of N
  benefits when you activate") aren't passive aggregations or session-long toggles → descriptive prose.
- **Targeted buffs on someone else.** Effects cast ON another creature (Cleric Blessing of the
  Trickster — advantage on another's Stealth) are out of scope for a single-character sheet → prose.
- **Battlefield geometry / encounter state.** Aura allies/enemies, cover, mounted/underwater combat,
  line-of-sight, and situational mid-combat advantage ("vs a Frightened target", "while you can't see
  the target") depend
  on transient geometry the passive aggregator (a property of the _character_, not the _encounter_) does
  not own. Auras are modeled INFORMATIONALLY (`aura` kind surfaces radius + effect, rolls/tracks
  nothing); only PERMANENT advantage clauses are modeled (`advantage-on`).
- **Item charge / duration / sentience timing.** The mechanical effect of a while-active item is
  modeled (Boots of Speed → `speed-multiplier`), but the time-box itself (the timer / per-use charge
  accounting) stays manual until a charge/duration-aware activation layer exists. A sentient item's own
  senses / Intelligence / personality are the _item's_ creature-like properties, not bonuses to the
  wielder → descriptive item text.
- **Hit-Dice-funded recovery.** Trackers restored by spending Hit Dice (some Berserker / Paladin
  variants) rather than a pool stay prose tied to the Hit-Dice pool.
- **Companion stat-block breadth.** Full creature stat-blocks (summons / Wild Shape forms / steel
  defender) are beyond the current engine breadth — tracked in `docs/AUTOMATION_BACKLOG.md`.
