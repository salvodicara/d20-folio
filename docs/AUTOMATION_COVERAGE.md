## Status legend

| Status        | Meaning                                                        |
| ------------- | -------------------------------------------------------------- |
| **automated** | Mechanics modeled as Grants the engine evaluates + renders.    |
| **partial**   | Some mechanics modeled, some still prose/manual.               |
| **override**  | No auto-compute; only a manual override / placeholder tracker. |
| **narrative** | Prose / roleplay; no numeric mechanic to model.                |

> **What "automated" means here, precisely.** A mechanic is _modeled_ iff it is a typed `Grant`
> the engine evaluates via `evaluateGrants`, or a `mechanics.tracker`/`actions`/`rider` the
> smart-tracker/compute layer surfaces (the engine never rolls dice â€” it shows FORMULAS, so a
> shown formula counts). **"Modeled" is not the same as the owner's interaction-cost bar for
> "automated"** (one tap; all consequences flow; override always available): a grant whose
> consumer is dead code, a rider computed but rendered nowhere, or an effect gated on a second
> manual toggle is modeled-but-not-automated. That gap frontier â€” and the seams that close it â€”
> is tracked in `docs/AUTOMATION_BACKLOG.md`; this matrix records the per-entity _modeled_ state.

> **Per-kind exposure is NOT tracked here.** Whether each `Grant` kind has a live consumer seam
> is code-enforced by `tests/unit/grant-kind-exposure.guard.test.ts` (every kind in the `Grant`
> union must register a verified consumer file+token, a justified PLUMBING entry, or a ranked
> OPEN item in the BACKLOG; adding a kind without registering it fails CI). That guard's table is
> the single source of truth for per-kind exposure â€” it is not duplicated in either doc.

### Backgrounds

Every 2024 background packs the same five benefits: an ability-score increase (+2/+1 or +1/+1/+1), 2 skill proficiencies, 1 tool proficiency, 1 Origin feat, and a starting-equipment package. The grant engine owns three of these: the 2 skills and 1 tool are emitted as `skill-proficiency` + `tool-proficiency` grants by `buildBackgroundGrants` (`src/data/backgrounds.ts:796`), routed into `evaluateGrants` via `resolveGrantSourcesForBackground` (`src/lib/resolve-grant-sources.ts:215`). The Origin feat reaches `evaluateGrants` through the features pipeline: `syncOriginFeats`/`getBackgroundOriginFeat` (`src/lib/character-build.ts:90`, `src/data/backgrounds.ts:865`) projects the resolved feat slug into `character.features`, so the feat's own grants aggregate normally. Every referenced feat slug resolves to a real feat in `feats.ts` â€” no dropped links. The ASI (`backgroundAsi`) and equipment (`resolveBackgroundEquipment`) are deliberately NOT grants: they are non-idempotent creation-owned snapshots (baked into stored scores / inventory at creation, with override), per the documented design in `docs/MECHANICS.md`.

All 4 SRD backgrounds below share the same coverage; Notes lists the Origin feat (the variable benefit) plus anything notable.
